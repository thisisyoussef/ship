How I actually ran the audit

This page is the companion to the formal submission. It explains the rationale behind the measurement choices, the tools I used, what each tool actually does, what I intentionally did not spend time on, and how I would defend the work if someone asked me to justify every decision in the audit.

My goal was not to look busy across the repo. My goal was to build a credible baseline with enough depth that every measurement could survive follow-up questions.

Tooling rationale

What the tools are and why I used them

TypeScript AST scan

What it is: a syntax-tree pass over TypeScript source that counts actual language constructs like any, as, and non-null assertions.
Why I used it: it is more trustworthy than raw grep because it distinguishes real syntax from comments, strings, and false positives.

Vite production build

What it is: the actual frontend production bundling step. It emits the files a browser would download in a real deployment.
Why I used it: bundle size claims should come from the production output, not from estimates based on dev-server behavior.

Source maps + depcheck

What it is: source maps map built code back to source modules; depcheck looks for declared dependencies that are not actually imported.
Why I used them: they reveal where the large bundle weight is coming from and whether some dependency cost is avoidable entirely.

ab for latency benchmarking

What it is: ApacheBench, a simple HTTP load generator that can hit one endpoint repeatedly at controlled concurrency.
Why I used it: I started with autocannon, but the dev rate limiter polluted results with 429s. ab plus API restarts gave me clean, repeatable local baselines.

PostgreSQL statement logs

What it is: the database logging every SQL statement and its execution duration.
Why I used it: it shows real query count and shape for user flows. That matters because query count and query shape are often more actionable than abstract ORM intuition.

Playwright + axe + Lighthouse

What they are: Playwright automates real browser behavior, axe is a rule-based accessibility scanner, and Lighthouse is a broad browser audit that includes accessibility scoring.
Why I used them together: no one of these tools gives the full truth. Playwright reveals behavior, axe catches concrete violations, and Lighthouse shows the score the team might otherwise over-trust.

Bundle size: what these tools actually tell me

Production output, not intuition

What Lighthouse does not tell me

Lighthouse is not the right first tool for understanding build composition. It can show symptoms of a heavy app, but it does not tell me which packages and chunks are responsible for the weight.

That is why I started with the actual Vite production build, then moved into sourcemaps and dependency usage.

Why I used sourcemaps after source-map-explorer failed

The explorer tool hit a sourcemap parsing issue. Instead of abandoning the analysis, I used the emitted source map directly to group major contributors. That was still enough to identify where the weight lives: editor and collaboration dependencies, plus a few large UI libraries.

API latency: why I chose these endpoints and this benchmark style

Real flows, controlled load

Why these five endpoints

  • /api/documents for broad workspace navigation
  • /api/issues for work tracking
  • /api/documents/:id for targeted reads
  • /api/weeks/:id/issues for week/sprint execution view
  • /api/search/learnings for retrieval/search behavior

That gives a balanced slice of list reads, detail reads, board-style reads, and search.

What P50, P95, and P99 mean

  • P50: the median experience
  • P95: the tail most users notice when a system starts feeling slow
  • P99: the worst edge of the distribution

I emphasized P95 because the assignment asks for realistic performance, and tail latency is where real user frustration starts to show up.

Database efficiency: what query logs give me that endpoint timing does not

Shape beats guesswork

What query count tells me

Query count answers a different question than endpoint latency. It tells me whether the app is inefficient because it is doing too many trips to the database, especially in list views where N+1 problems usually hide.

What query shape tells me

Query shape tells me whether the app is leaning on broad filters, association-heavy joins, or workspace-wide sorting. In a codebase built around a unified document table, that matters a lot.

QuestionAnswer from the auditWhy I care
Is there obvious N+1?No in the sampled flows.This means the app is not wasting round trips in the most common list and board views I checked.
Are query counts high?No. The audited flows stayed between 4 and 5 statements.This means the bigger risk is probably broad unified-model work, not raw query explosion.
What is the likely scale pressure point?Workspace-wide document reads and association-heavy joins.That matches the slowest API route and the architecture’s document graph design.

Testing: why raw test count is not enough

Harness quality vs. suite health

What I was trying to separate

I wanted to separate “does the repo have a good testing architecture?” from “is the current suite actually healthy?” Those are different questions, and Ship is a good example of why that distinction matters.

Harness strongBaseline not clean

Why I did not stop at total tests

A high test count can still hide the wrong gaps. What matters is whether the tests cover the places where the product is most likely to fail in a meaningful way. In Ship, that means collaboration recovery, multi-user behavior, and high-friction weekly workflows.

Runtime behavior: why manual testing still matters

There are things a test suite will not tell me

What manual runtime probing gave me

Manual probing surfaced two different kinds of truth. First, it showed that the collaboration layer is stronger than the red test baseline might imply. Second, it revealed a class of confusing, non-catastrophic failure that automated tests often miss: interaction friction that makes the product feel broken before it is technically broken.

Why offline and concurrent edits mattered here

Ship’s architecture makes real-time editing a first-class feature. If I skipped offline and concurrent edit tests, I would be ignoring one of the most architecture-specific risks in the whole repo.

Accessibility: what Lighthouse is, and why I paired it with axe

This is where one-tool thinking fails hardest

What Lighthouse is

Lighthouse is a broad browser audit built into Chromium-based tooling. It gives high-level scores across categories like performance, best practices, SEO, and accessibility. It is useful because it is fast, familiar, and easy to compare across pages.

But Lighthouse is still a score layer. It is not a complete accessibility truth engine.

What axe is

axe is a rule-based accessibility engine that checks the page for concrete WCAG-related violations such as contrast failures, semantic misuse, missing relationships, and form/control problems. It is much better at telling me what is broken.

axe gives me violations. Lighthouse gives me a score. I need both to avoid fooling myself.

How I would defend the work in conversation

The rationale in plain language
Why didn’t you fix issues during the audit?Because the assignment explicitly treats the audit as a hard diagnostic gate. Fixes would blur the baseline and make it harder to separate measurement from intervention.
Why not benchmark every endpoint?I wanted depth on representative, user-facing flows rather than shallow numbers on everything. Tracing real browser requests let me pick a defensible sample tied to actual product usage.
Why did you trust your type-safety counts?I used a TypeScript AST scan rather than plain text search, which gave me syntax-aware counts. That is a stronger baseline than grep for a language-level audit.
Why do you keep emphasizing P95?P95 is where users feel the system getting slow. Median latency can flatter a route that still produces bad tails under load.
Why is the accessibility section so critical if Lighthouse scored 100?Because a perfect score did not match the actual violations found by axe or the keyboard friction observed in the browser. The contradiction is the point.
If someone says the database looks fine because queries are fast locally, what would you say?I would agree that local query count is healthy, but I would point to the architecture: the broad document list path and association-heavy reads are still the likely first database-linked scale pressure points.
The shortest version of the whole audit

Ship is architecturally coherent. The unified document model, collaboration stack, and test harness design are real strengths. The biggest current weaknesses are practical, not philosophical: the main document-list path is the backend bottleneck, the frontend initial payload is too heavy, route-level type discipline is loose in important places, accessibility confidence is overstated if you rely on Lighthouse alone, and the automated test baseline is not clean enough around the most product-specific behaviors.
