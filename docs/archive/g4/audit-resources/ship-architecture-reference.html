1 — Repository Overview

Phase 1 repository overview: current package mental model

The repository is structured as a monorepo with three primary package boundaries: `web/`, `api/`, and `shared/`. The intended shape is consistent with the architecture docs: the web app renders and edits work, the API owns persistence and collaboration, and the shared package holds the type system that keeps one document model usable on both sides of the boundary.

Frontend package

web/

React application, page shell, editor UI, cached reads, and the client side of collaboration and work management flows.

Backend package

api/

Express API, database access, relationship handling, visibility checks, and the server-authoritative source of truth.

Contract package

shared/

Shared document, relationship, and property types that let the product keep one model without losing all type-level structure.

`web/` talks to `api/` through HTTP plus real-time collaboration channels.
`web/` and `api/` both depend on `shared/` so the document model is not duplicated in two incompatible ways.
2 — Data Model

Data model: current implementation shape

The current schema is in `api/src/db/schema.sql`. The migration trail that explains the relationship model is in `api/src/db/migrations/020_document_associations.sql`, `027_drop_legacy_association_columns.sql`, `029_drop_program_id_column.sql`, `033_sprint_to_week_rename.sql`, and `037_week_dashboard_model.sql`. The application-level type mirror lives in `shared/src/types/document.ts`. Taken together, they show a clear design: a central `documents` table, a separate `document_associations` table for most membership-style relationships, and a TypeScript `belongs_to` shape that surfaces those relationships back to the client.

Current model at a glance

`documents`

Stores the core identity and content for wiki pages, issues, programs, projects, sprints, person docs, plans, retros, standups, and weekly reviews.

id workspace_id document_type content properties parent_id

`document_type`

Acts as the discriminator for queries and behavior. Routes filter the same table by type instead of querying separate issue/project/week tables.

issue project sprint weekly_plan weekly_review

`document_associations`

Handles program, project, sprint, and some parent relationships using `document_id`, `related_id`, and `relationship_type` rather than legacy direct FK columns.

program project sprint parent
Table Purpose Important relationships
workspaces Top-level tenant boundary and week cadence anchor (`sprint_start_date`). Referenced by users, memberships, sessions, documents, tokens, files, comments, iterations, and audit logs.
users Global identity records across workspaces. Referenced by memberships, sessions, invites, tokens, documents (`created_by`), comments, iterations, and audit logs.
workspace_memberships Authorization layer for who can access a workspace. Links `users` to `workspaces`; explicitly separate from person documents.
documents Core content and work table for nearly every first-class artifact in the product. Self-references through `parent_id`; referenced by associations, history, snapshots, links, comments, and iterations.
document_associations Junction table for membership-style document relationships. Both `document_id` and `related_id` point to `documents`; typed by `relationship_type`.
document_history Audit trail of field changes on documents. `document_id -> documents.id`.
document_snapshots Saved document state for conversions and undo-like flows. `document_id -> documents.id`.
document_links Backlink-style graph between documents. `source_id` and `target_id` both reference `documents`.
comments Threaded inline document comments. `document_id -> documents.id`, optional self-reference via comment `parent_id`.
sprint_iterations / issue_iterations Progress telemetry for weeks and issues. Reference the relevant document row plus workspace and author.
files Attachment metadata. References workspace and uploader; file bytes live outside the DB.
sessions, oauth_state, api_tokens, audit_logs, workspace_invites Security, auth, and compliance support tables. Orbit around users and workspaces rather than the document graph.

How one table serves docs, issues, projects, and sprints

All of them share the same base fields: workspace, title, content, collaboration state, lifecycle timestamps, and visibility. Type-specific data lives in `properties`, while the code branches on `document_type` to interpret that row as an issue, project, week, plan, or review.

What the discriminator is

`document_type` is a PostgreSQL enum and a shared TypeScript union. The routes use it directly in queries like `WHERE d.document_type = 'issue'` or `WHERE d.document_type = 'sprint'` rather than reading from specialized tables.

How relationships actually work

The current relationship mechanism is `document_associations`, exposed to the frontend as a `belongs_to` array with types `program`, `project`, `sprint`, and `parent`. Creation and update routes insert and delete association rows rather than mutating legacy foreign-key columns.

Important nuance

`parent_id` still exists as a direct self-reference on `documents` and is used for true hierarchy and some child-content flows, while `document_associations` handles most membership and grouping relationships. The model is unified, but not reduced to one relationship mechanism.

-- From api/src/db/schema.sql
CREATE TYPE document_type AS ENUM (
  'wiki', 'issue', 'program', 'project', 'sprint',
  'person', 'weekly_plan', 'weekly_retro', 'standup', 'weekly_review'
);

CREATE TABLE documents (
  id UUID PRIMARY KEY,
  workspace_id UUID NOT NULL,
  document_type document_type NOT NULL,
  title TEXT NOT NULL,
  content JSONB,
  yjs_state BYTEA,
  parent_id UUID REFERENCES documents(id),
  properties JSONB DEFAULT '{}'
);

CREATE TABLE document_associations (
  document_id UUID NOT NULL REFERENCES documents(id),
  related_id UUID NOT NULL REFERENCES documents(id),
  relationship_type relationship_type NOT NULL
);
-- From api/src/routes/issues.ts and api/src/routes/documents.ts
SELECT *
FROM documents d
WHERE d.workspace_id = $1
  AND d.document_type = 'issue';

INSERT INTO document_associations (document_id, related_id, relationship_type)
VALUES ($1, $2, $3)
ON CONFLICT (document_id, related_id, relationship_type) DO NOTHING;

-- From shared/src/types/document.ts
export type BelongsToType = 'program' | 'project' | 'sprint' | 'parent';

3 — Request Flow

High-level request and collaboration flow

The docs and route layer support a strong runtime model: the React app loads lists and metadata through REST and cached reads, opening a document restores local editor state first, the editor then rejoins live collaboration through WebSocket and Yjs, and the API persists both collaborative state and structured document data to PostgreSQL. Weekly accountability logic then sits on top of the same shared model.

Step 1

React app opens

The user lands in the web application shell.

Step 2

REST + cache

Lists and metadata come from REST endpoints and client cache.

Step 3

IndexedDB restore

Document editing starts with local editor state for fast open.

Step 4

WebSocket + Yjs

Realtime collaboration syncs and merges concurrent edits.

Step 5

API persistence

The backend writes Yjs state, JSON content, and extracted properties.

Step 6

Workflow layer

Planning and accountability logic evaluate the same document model.
