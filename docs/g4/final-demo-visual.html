1799070api/src/routes/weeks.ts

The sprint-board route stopped round-tripping through two separate queries

Beforeverify, then fetch
const { isAdmin } = await getVisibilityContext(userId, workspaceId);
const sprintResult = await pool.query(`SELECT d.id ...`);
if (sprintResult.rows.length === 0) {
  res.status(404).json({ error: 'Week not found' });
  return;
}
const result = await pool.query(`SELECT d.id ... JOIN sprint_da ...`);
Afterone access-aware query
const isAdmin = req.isSuperAdmin === true
  ? true
  : (await getVisibilityContext(userId, workspaceId)).isAdmin;
const result = await pool.query(`WITH accessible_sprint AS (...)
  SELECT sprint.id as sprint_id, issue.id, ...
  FROM accessible_sprint sprint
  LEFT JOIN LATERAL (...) issue ON TRUE`);

What / How / Why / Result

What
Collapsed sprint verification and issue fetch into one route-level statement.
How
`WITH accessible_sprint` + `LEFT JOIN LATERAL` preserved both `404` and empty-array behavior.
Why
No N+1 was present, so removing redundant round trips was the highest-value query-efficiency fix.
5 queries → 3 load sprint board • 40% fewer queries • option A met
